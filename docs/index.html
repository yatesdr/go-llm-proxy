Make any model work with any client

A single-binary LLM proxy that translates between API protocols, routes requests across backends, and runs upstream tools locally. One YAML config. No dependencies.

Download View Source

Problems this solves

Coding assistants don't speak your model's language

Claude Code uses the Anthropic Messages API. Codex uses the OpenAI Responses API. Your vLLM instance speaks Chat Completions. The proxy translates between them automatically.

Models spread across machines and providers

Local GPU servers, cloud APIs, third-party providers — each with different URLs and auth. The proxy aggregates them behind one endpoint. Clients see one model list.

Server-side tools don't exist locally

When Claude Code requests web search or sends images to a text-only model, local backends can't handle it. The proxy intercepts these, executes them (Tavily search, vision model description), and injects the results.

No visibility into who's using what

Per-request logging with token counts, latency, and per-user breakdowns. Web dashboard and CLI reports. SQLite — no external database needed.

Get started

Create a config.yaml with your models and keys:

listen: ":8080" models: - name: my-model backend: http://192.168.1.10:8000/v1 keys: - key: sk-your-secret-key name: admin

Run it:

# Docker docker run --rm -p 8080:8080 -v ./config.yaml:/config/config.yaml:ro ghcr.io/yatesdr/go-llm-proxy:latest # Or binary ./go-llm-proxy -config config.yaml

Point any OpenAI or Anthropic-compatible client at http://localhost:8080/v1 with your proxy API key. The config reference covers all options.

Features

Protocol translationAnthropic Messages API and OpenAI Responses API translated to Chat Completions. Any client works with any backend.
Model multiplexingRoute multiple models across multiple backends. Name rewriting, per-model timeouts, auto protocol detection.
API key managementIssue proxy keys with per-key model restrictions. Backend credentials stay on the server. Supports Bearer and x-api-key headers.
Vision pipelineImages sent to text-only models are described by a vision-capable model and replaced with text. Transparent to the client.
Web searchServer-side search tools (Claude Code, Codex) executed at the proxy via Tavily. Results injected into the conversation.
Usage monitoringPer-request logging to SQLite. Web dashboard, CLI reports, per-user/model breakdowns.
Config generatorBuilt-in web UI creates ready-to-use configs for Claude Code, Codex, OpenCode, and Qwen Code.
Hot reloadConfig reloads on file save. Add models or rotate keys without restarting.
Security hardenedConstant-time auth, rate limiting, SSRF protection, sanitized error responses, path allowlisting.

Supported backends

Anything that speaks OpenAI or Anthropic protocol works. Tested with:

vLLM llama-server Ollama OpenAI API Anthropic API MiniMax Zhipu (GLM)

How it works

Client request (any protocol) → Protocol handler parses request → Translate to Chat Completions if needed → Pipeline: describe images, inject web search tool → Send to backend → Pipeline: execute web search if called, re-send → Translate response back to client protocol → Stream to client

The pipeline is optional. Without processors configured, the proxy just translates and routes. With it, images and search work transparently on text-only backends.
